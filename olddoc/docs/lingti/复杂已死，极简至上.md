# 复杂已死，极简至上

---

## 一月，ClawdBot

2026 年 1 月，我第一次打开 ClawdBot（现在叫 OpenClaw）。

说实话，是带着期待的。一个能帮你操控电脑、对接各种服务的 AI Bot，听上去挺像那么回事。

然后我发现——它不支持企业微信、飞书、钉钉、微信公众号。我每天打开的工作软件，它一个都不认识。我们公司自己也做互联网产品，也没法对接上去。

演示视频里，Bot 在 Slack 里如鱼得水、在 Discord 里谈笑风生。但对我来说，跟我没关系。远程帮你点外卖？连美团都进不去，点什么外卖。

关掉页面，没再打开。

---

## 不只是我一个人的问题

后来我加了几个 Clawd 的交流群，本来想看看有没有人找到变通办法。

结果满屏都是抱怨。

"企业微信能接吗？" "飞书怎么对接？" "钉钉有人搞定了吗？"

一条又一条，一天又一天，同样的问题，没有答案。

这不是我一个人的问题。这是所有中国 Bot 爱好者、开发者的问题。不是我们能力不够，是没人给我们造那把钥匙。

我在群里潜水了大约一周，看着一个又一个人带着热情来、带着失望走。

---

## 1 月 31 日，动手

感受了大概一周大家的痛苦之后，1 月 31 日，我打开了 Vim。

没人做，那就我来。

从企业微信开始。它的接入是出了名的麻烦——公网服务器、备案域名、HTTPS 证书、回调服务……光前置条件就劝退一大片人。

我盯着企业微信的文档看了很久，想到了一个办法：**云中继**。

云端只做消息转发，AI 处理全在本地。不要公网服务器，不要域名备案，不要 HTTPS 证书。一条 WebSocket，把企业微信和你的电脑直接连起来：

```
企业微信 --> keeper.kayz.com(云中继) --WebSocket--> 你的电脑(本地AI)
```

第一次在企业微信里发消息，看到本地 Bot 秒回的时候，我知道路走对了。

后来查了很多资料，这个方案市场上确实没有。于是我逐渐把这些方案整合到一个代码库里，开源出来。

## 复杂已死

做的过程中，我也认真研究了 OpenClaw。

我不喜欢它的复杂。npm install 下来几百兆 node_modules，各种依赖、各种配置、各种运行时……看着它的安装指南，我想起了 Multics。

上世纪六七十年代，MIT、贝尔实验室、通用电气联手搞 Multics，投了大量资金和顶尖人才。设计宏大，功能丰富，架构复杂。贝尔实验室最终因为复杂性失控退出了这个项目。

从 Multics 项目离开的 Ken Thompson 和 Dennis Ritchie 做了个反方向的东西：UNIX。极致简洁。早期实现里搜索就是暴力 for loop，调度器就是最朴素的轮转。没有花哨的优化，没有宏大的架构。

UNIX 活了下来，改变了世界。

简洁意味着几乎不犯错，意味着可以演进，意味着任何人都能理解、修改、信任它。

**复杂已死，我信仰简单。**

lingti-bot 编译出来就是一个 30MB 的文件。没有 node_modules，没有 requirements.txt，没有 Docker，没有数据库。`scp` 到任何机器上就能跑。

```bash
# 这就是全部的安装过程
curl -fsSL https://coco.kayz.com/install.sh | bash -s -- --bot
```

一行命令，结束。

## 为什么是 Go

我做了大量网络底层高性能、高并发研发，2020 年用 Go 打造了 App Store 中国畅销榜第 13 位的产品——灵缇游戏加速。除了 UX 以外，几乎所有全栈开发都用 Go 完成。

游戏加速是对网络编程要求最苛刻的场景之一。每秒数万级的 packets 处理是家常便饭，延迟多几毫秒玩家就能感知到，丢一个包可能就是一次卡顿。在这种环境下写了几年代码，我对 Go 的能力边界摸得很清楚：它不是什么都好，但在高吞吐、高并发的网络 IO 场景下，没有比它更趁手的工具了。

Go 的厉害在于它两头都能够到。往上，几十行代码就能撑起一个高性能的网络服务，goroutine 开几千个跟玩一样，不用操心线程池、不用回调地狱；往下，它能直通系统级——操作内存、控制进程、调用系统调用，这些事很多高级语言根本碰不到。JavaScript 再怎么折腾，碰到 raw socket 和内存操作就得绕路。Python 更不用说。

微软去年宣布把 TypeScript 编译器用 Go 重写，Anders Hejlsberg 亲自操刀，性能提升 10 倍。团队试过好几种语言做原型，最后选了 Go——因为 Go 的代码结构跟现有 JavaScript 实现最接近，迁移成本最低，同时又能编译成原生机器码，自带 GC 和并发原语。换句话说，Go 是唯一一个让你"写着像高级语言，跑起来接近系统语言"的选项。

这也是我在游戏加速中体会最深的一点。Go 编译出来的 binary 是真正的原生可执行文件，不需要任何运行时，性能直逼 C 级别，同时又自带 goroutine 这套并发模型。一个 30MB 的 binary 就能扔到树莓派、路由器、NAS 上直接跑——不像 Node.js 需要先装个几百兆的运行环境。

而 AI Bot 的未来场景只会比游戏加速更极端：动辄百万级的 Token 在网络上传输，几十乃至上百个 AI 后端组成的 Agents 集群在同一个框架下协作，每个请求都是长连接、大 payload、高并发——这跟游戏加速的本质是一样的，都是在考验你能不能在高压下把每一个字节、每一毫秒都管好。

大吞吐、高并发、嵌入式部署，三个方向 Go 全吃。lingti-bot 一次编译，同一份代码出 macOS、Linux、Windows、ARM、MIPS 的二进制，到处执行。

如果要做下一代 Bot 框架，我认为目前最适合的语言就是 Go。这不是信仰，是数十万行 Go 代码写出来的判断。

## 极速接入

国内 Chat 平台的认证和接入机制跟国外很不一样。企业微信要验证域名、要配 IP 白名单，微信公众号要备案、要 HTTPS——这些门槛把大量想玩 Bot 的人挡在了外面。

我做了云中继，就是为了把这些门槛踩平。不需要你有服务器，不需要你懂运维，一条命令接进去，5 分钟搞定。

这也是 lingti-bot 跟 OpenClaw 最本质的区别之一：OpenClaw 对中国平台的本地化支持几乎为零，而 lingti-bot 从第一天起就是为这些平台写的。

## 为什么叫灵小缇

灵缇犬，Greyhound，世界上跑得最快的狗，时速能到 72 公里。流线型的身体，没有一两多余的肉，每一块肌肉都是为速度长的。

但灵缇犬真正打动我的不只是快。它是少数不靠体型、不靠攻击力立足的犬种——瘦长、安静、优雅，不叫不闹，平时就趴在你脚边。但一旦需要，六步之内加速到极限。

这就是我想要的 Bot 的样子。不臃肿，不炫技，平时安安静静地待着，占用极少的资源。但你一条消息过来，它就以最短路径、最快速度把事情办了。

灵缇游戏加速也是同一个"灵缇"。从加速器到 Bot，名字没变，理念也没变：**用最精简的形态，做最快的事。**

从 1 月 31 日敲下第一行代码到现在，每个功能、每行代码都指向同一个方向——

> **极简至上，效率为王，一次编译，到处执行，极速接入。**

没人给我们造钥匙，那就自己造一把。
